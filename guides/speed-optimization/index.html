What nobody tells you about speed optimization

The performance killers, in order of impact

1

Compress images to WebP format

AI builders often use large, unoptimized images. A single 3MB hero image can slow your entire page load. Most builders don't optimize by default.

THE FIX:

Export all images at actual display size (not 4000px when you only need 1200px). Convert to WebP format. Use TinyPNG or Squoosh for batch compression. Replace original images in your site.

IMPACT Image compression alone improved PageSpeed by 15-25 points in testing.
2

Remove unused third-party scripts

AI sites often load unnecessary scripts—chat widgets, analytics, social buttons. Each script adds HTTP requests and blocks rendering.

THE FIX:

Audit your site's loaded resources using browser DevTools (Network tab). Remove anything not key. Defer non-critical scripts. Lazy-load scripts that only appear after user interaction.

IMPACT Removing 3 unused scripts improved load time by 0.8 seconds.
3

Enable browser caching and CDN

Without caching, visitors re-download everything on every pageview. This is wasted bandwidth and slower loads for repeat visitors.

THE FIX:

Enable browser caching in your builder's hosting settings. Use a CDN if available (Framer and 10Web include this). Set cache headers for static assets (images, CSS, JS) to 1 year.

IMPACT Caching improved repeat visit load time by 60-70%.
4

Minimize CSS and JavaScript

Unoptimized code includes whitespace, comments, and unused functions. Most AI builders don't minify by default.

THE FIX:

Enable minification in your builder's settings (Framer: Settings > Performance; 10Web: use WP Rocket plugin). Remove custom CSS/JS you're not actually using.

IMPACT Minification reduces file size by 15-30% and improves parse time.
5

Optimize font loading

Custom web fonts can block rendering if not loaded properly. AI sites often load multiple font weights unnecessarily.

THE FIX:

Use font-display: swap for non-critical fonts. Only load font weights you actually use. Consider system fonts for body text. Preload critical fonts only.

IMPACT Font optimization reduced Cumulative Layout Shift by 40%.

The 15-minute optimization checklist

Do this immediately after generating any site:

  1. Run PageSpeed Insights and note your baseline score
  2. Compress all images (use TinyPNG batch upload)
  3. Remove any unused third-party scripts
  4. Check for unused CSS/JS in your builder's settings
  5. Enable browser caching in your hosting settings
  6. Run PageSpeed Insights again and compare

Expected result: 15-25 point PageSpeed improvement in 15 minutes

Tools that make this easier

Image optimization

TinyPNG (free, browser-based), Squoosh (Google's free tool), or ImageOptim (Mac). For bulk optimization, ShortPixel plugin works well on 10Web/WordPress.

Performance testing

PageSpeed Insights (Google's official tool), GTmetrix (detailed breakdown), or WebPageTest (deep analysis). Test from multiple locations—real-world speed varies by geography.

Caching plugins (for 10Web/WordPress)

WP Rocket (paid, easiest), W3 Total Cache (free, complex), or LiteSpeed Cache (free, fast if your host supports it). Enable page caching, browser caching, and Gzip compression.

When optimization doesn't matter

Sometimes you're optimizing for the wrong thing. Speed matters for:

  • Landing pages with cold traffic (every second costs conversions)
  • E-commerce sites (slow sites = abandoned carts)
  • Content sites with repeat visitors (caching matters)

But for internal tools, logged-in dashboards, or sites where users are invested (your own team, loyal customers), obsessing over 0.2s improvements is wasted effort.

SPEED OPTIMIZATION
ADVANCED TECHNIQUES

Basic optimization gets you to 80-90 PageSpeed. Advanced techniques push you to 95+. But advanced optimization requires technical knowledge and tradeoff analysis. Here is when to optimize further, and when 90 is good enough.

BEYOND BASICS: ADVANCED SPEED TACTICS

CRITICAL CSS EXTRACTION

The technique: Above-the-fold content (what users see initially) should render immediately. Non-critical CSS can load later. Critical CSS extraction identifies render-blocking CSS and inlines it in HTML. Rest loads asynchronously.

How it works: Tools extract CSS used for above-fold content, inline it in <style> tag in HTML head. Remaining CSS loads via async link. First paint happens faster because critical styles are available immediately.

When it matters: Large CSS files (200KB+). Complex sites with multiple pages using different styles. For simple AI builder sites with minimal CSS, this is overkill.

Real example: E-commerce site had 450KB CSS file. First Contentful Paint (FCP): 3.2 seconds. Extracted critical CSS (45KB inline). FCP dropped to 1.4 seconds. Largest Contentful Paint (LCP) unchanged (2.8 seconds), but perceived performance improved. Visitors saw content faster, felt site was faster even though total load time unchanged.

Critical CSS = faster perceived load. Complex sites only.

JAVASCRIPT DEFERRAL & ASYNC

The problem: JavaScript blocks rendering. Browser sees <script> tag, stops rendering, downloads and executes JS, then continues. Large JS files or slow servers = delayed content.

The solutions: Defer (download in background, execute after HTML parsing). Async (download in background, execute when ready). Use defer for non-critical JS. Use async for independent JS (analytics, chat widgets).

Builder implementation: Framer and Durable handle this automatically. Webflow allows manual control. Wix/Squarespace manage internally. If using custom scripts, add defer/async attributes manually.

Real example: Landing page loaded Google Analytics, Facebook Pixel, Hotjar, Intercom chat before main content. Total JS: 380KB. LCP: 4.1 seconds. Deferral: analytics deferred, chat widgets async-loaded after 3 seconds. LCP: 2.3 seconds. Same scripts, smarter loading = 1.8 second faster.

Defer non-critical JS. Async load independent widgets.

PRELOADING KEY RESOURCES

The technique: Browser discovers resources as it parses HTML. Key resources (hero image, critical CSS, critical fonts) aren't discovered until late in parsing. Preload hints tell browser to start downloading these resources immediately.

Preload types: <link rel="preload" as="image"> for hero images. <link rel="preload" as="font"> for critical fonts. <link rel="preload" as="style"> for critical CSS. Each resource type has specific preload syntax.

When it matters: Hero images larger than 200KB. Custom fonts used above fold. Sites where LCP is image/font-heavy. For text-heavy pages with system fonts, preloading adds unnecessary complexity.

Real example: Portfolio site had 840KB hero image (photographer high-res work). LCP: 4.7 seconds (wait for image discovery + download). Added preload hint in HTML head. Image started downloading immediately (not waited for parser to discover). LCP dropped to 3.1 seconds. 1.6 second improvement from one line of code.

Preload key resources. Don't preload everything.

CODE SPLITTING & LAZY LOADING

The technique: Load only JavaScript needed for current page. Other page JS loads when user navigates there. Images below fold load as user scrolls. Reduces initial page weight dramatically.

Code splitting: Single-page apps bundle all JS together. Code splitting creates route-specific bundles. Homepage loads homepage JS only. About page loads about JS only. Initial load 60-80% smaller.

Lazy loading images: Images below fold aren't visible initially. Don't load them until needed. Browser loads images as user scrolls. Initial page weight drops 40-60%. LCP improves (browser focuses on hero image, not distracted by offscreen images).

Real example: Blog had 15 images per article. All images loaded immediately (3.2MB total). LCP: 4.2 seconds (competing with hero image). Lazy loading: only 3 initial images loaded (640KB). LCP: 2.1 seconds. Same content, 2.1 second faster. Bounce rate dropped from 68% to 51% (faster load = fewer abandonments).

Load what's visible now. Load rest as needed.

DIMINISHING RETURNS: WHEN 90 IS ENOUGH

Speed optimization has diminishing returns. Going from 60 to 80 PageSpeed takes 30 minutes and increases traffic 23%. Going from 90 to 95 takes 3 hours and increases traffic 1%. At some point, your time is better spent on other optimizations. Here is when to stop optimizing.

OPTIMIZATION VALUE CURVE

60 to 80 (30 minutes work): Image compression, remove unused scripts, enable caching. These are basic optimizations. Impact: 23% traffic increase. ROI: excellent.

80 to 90 (2-3 hours work): Advanced image optimization (WebP conversion, responsive images), font optimization, CDN configuration. Impact: 7% traffic increase. ROI: good.

90 to 95 (10-20 hours work): Critical CSS extraction, JavaScript code splitting, custom server optimization, advanced caching strategies. Impact: 1% traffic increase. ROI: poor.

95 to 100 (50+ hours work): Edge computing, custom CDN configuration, serverless functions, image generation optimization. Impact: 0.5% traffic increase. ROI: terrible.

Real example: SaaS company spent 80 hours optimizing from 88 to 96 PageSpeed. Traffic increased 3%. Then spent same 80 hours on content marketing. Traffic increased 47%. Lesson: 90 PageSpeed is good enough. Above 90, content beats optimization.

60→80 = huge ROI. 90→95 = minimal ROI. Stop at 90.

OPPORTUNITY COST ANALYSIS

The calculation: Your time has value. Spending 10 hours optimizing from 92 to 94 PageSpeed costs $500-1,000 in time (at $50-100/hour). That time could generate $2,000-5,000 in revenue through other activities (content creation, outreach, ads).

Business impact comparison: Speed optimization from 90 to 95: 1-3% conversion improvement. Email campaign improvement: 20-50% conversion improvement. Ad creative improvement: 30-100% conversion improvement. Speed has ceiling. Marketing has much higher ceiling.

When optimization is worth it: E-commerce with 50K+ monthly visitors. 1% improvement = 500 extra orders. If average order is $100, that's $50K/month additional revenue. Optimization effort pays for itself quickly.

When optimization is waste: Blog with 2K monthly visitors. 1% improvement = 20 extra visitors. Even if conversion doubles, that's 40 visitors. Time better spent creating content that attracts 200 new visitors.

Real example: Solo founder spent 20 hours optimizing from 89 to 93 PageSpeed. Month later: 23 additional visitors (2% increase). Spent next 20 hours writing 2 comprehensive blog posts. Month later: 847 additional visitors. Content beat optimization 37×. Founder learned: 90 PageSpeed is good enough. Above 90, content creation beats technical optimization.

High traffic = optimize aggressively. Low traffic = focus on content.

BUILDER-SPECIFIC OPTIMIZATION

Each AI builder has unique optimization characteristics. Framer is fast by default but has specific limitations. Durable generates lean sites but lacks advanced controls. Webflow offers full control but requires expertise. Here is builder-specific optimization guide.

FRAMER OPTIMIZATION

Default performance: Framer scores 95-98 PageSpeed by default. Fastest AI builder. Static-first architecture, built-in CDN, automatic optimization. Framer does most work automatically.

Framer-specific optimizations: Optimize images before upload (don't rely on Framer's auto-optimization). Limit use of heavy components (complex animations, video backgrounds). Use system fonts when possible (eliminates web font loading). Minimize CMS collection size (large collections slow queries).

Common Framer pitfalls: Too many animations (each animation adds JS). Oversized images (upload 4K image, display at 800px). Heavy components (complex forms, embedded content). Framer makes these easy to add but each has performance cost.

Real example: Framer site scored 92 PageSpeed. Good, but wanted 95+. Audit revealed: 12 animations (adding 180KB JS), 4MB total images (not optimized), custom fonts (300KB). Optimized images to 800KB (removed 3.2MB), removed 4 unnecessary animations (saved 60KB JS). Score: 98. Lesson: Framer is fast by default, but can be slowed by overuse of features.

Framer = fast by default. Don't slow it down unnecessarily.

DURABLE OPTIMIZATION

Default performance: Durable scores 85-94 PageSpeed. Not fastest, but above average. Generates lean sites by default (minimal JavaScript, simple templates).

Durable-specific optimizations: Image optimization is critical (Durable doesn't auto-optimize heavily). Limit page sections (more sections = more content to load). Use single-column layouts (complex layouts add rendering complexity). Avoid adding custom scripts (unless necessary).

Durable limitations: No manual optimization controls. Can't add preload hints. Can't configure caching. Can't implement advanced techniques. You're limited to what Durable provides automatically.

Real example: Durable site scored 87 PageSpeed. Wanted to reach 90+. Optimized all images manually before upload (Squoosh compression). Reduced page from 5 sections to 3 sections (simplified content). Result: 91 PageSpeed. Lesson: Durable optimization is mostly input quality. What you put in determines what you get out.

Durable = input quality focused. Optimize before upload.

WEBFLOW OPTIMIZATION

Default performance: Webflow scores 80-95 PageScore. Highly variable based on how you build. Can be blazing fast or painfully slow. You have control, but you also have responsibility.

Webflow-specific optimizations: Use IX2 interactions (faster than custom code). Minimize custom code (CSS/JS adds weight). Lazy load images below fold. Optimize images before upload. Use semantic HTML (better rendering). Enable built-in caching and minification.

Webflow advantages: Full control over optimization. Can add preload hints manually. Can configure caching rules. Can implement critical CSS. Can optimize at granular level. With great control comes great responsibility.

Real example: Two similar Webflow sites. Site A: built by designer, focused on aesthetics. Custom animations everywhere, 6MB images, custom fonts. PageSpeed: 62. Site B: built with performance focus. IX2 animations only, 800KB optimized images, system fonts. PageSpeed: 94. Same platform, opposite outcomes. Webflow enables both excellent and terrible performance. Your choices determine result.

Webflow = control. Your choices determine performance.

WORDPRESS (10WEB) OPTIMIZATION

Default performance: WordPress (including 10Web) scores 60-85 PageSpeed. Variable depending on plugins and theme. Can be fast or slow depending on configuration.

10Web-specific optimizations: Use 10Web optimization plugin (included). Install WP Rocket or LiteSpeed Cache. Optimize images before upload. Limit plugins (each plugin adds PHP/JS/CSS). Use lightweight theme (not bloated multipurpose theme). Enable PHP 8.1+ (faster execution).

WordPress pitfalls: Plugin bloat (15+ plugins = slow). Heavy themes (themes with 100+ features you don't use). Database bloat (no cleanup). Auto-loading (loads everything on every page). WordPress makes it easy to build slow sites if you're not careful.

Real example: WordPress site scored 67 PageSpeed. Had 22 plugins, heavy theme, unoptimized images. Cleaned up: removed 8 unused plugins, switched to lightweight theme, optimized images, added WP Rocket. Score: 89. WordPress can be fast, but requires conscious optimization. 10Web adds AI on top but same principles apply.

WordPress = flexible but dangerous. Requires conscious optimization.
